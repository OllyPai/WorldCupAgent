## 一、项目范围

项目实现“基于大模型的工具调用智能体——世界杯竞赛播报助手”。系统需要体现“理解任务—选择工具—调用工具—观察结果—生成回答”的完整流程。

基础必做工具为：

1. 赛程查询；
2. 球员数据查询；
3. 比赛详情查询，比分属于比赛详情的一部分。

当前版本已在三类基础能力上扩展到七个工具，新增球员排行、队内最佳射手、门将扑救和射手榜前十查询。积分榜仍属于可选扩展，不影响基础版本验收。

课程要求中的直接回答、单工具调用、多工具调用、多轮对话、错误输入、异常提示、Web 可视化界面和不少于 10 条测试用例均属于必做内容。

## 三、接口约定

### 1. 工具向 Agent 的交付

李文杰向郑梓湃交付可直接调用的函数。每个工具至少提供稳定的函数名、参数说明和结构化返回值。

最小返回结构：

```python
{
    "success": True,
    "data": {},
    "error": None,
}
```

查询不到数据或调用失败时：

```python
{
    "success": False,
    "data": None,
    "error": "可供程序判断和用户理解的错误说明",
}
```

可预期的查询失败应返回结构化错误，不应直接导致整个应用崩溃。工具是否直接封装为某个 Agent 框架的工具对象暂不确定，由郑梓湃在集成层适配。

### 2. Agent 向前端的交付

Agent 提供一个统一入口。其具体表现形式可以是 Python 函数或 Web API，但核心输入输出保持一致。

输入：

```python
user_input: str
history: list[dict]
```

`history` 由前端维护，基本格式为：

```python
[
    {"role": "user", "content": "巴西队赛程"},
    {"role": "assistant", "content": "查询结果……"},
]
```

输出：

```python
{
    "answer": "面向用户的最终回答",
    "tool_calls": [
        {
            "tool": "工具名称",
            "input": {},
            "status": "success",
            "summary": "结果摘要",
        }
    ],
    "error": None,
}
```

发生模型、网络或系统级错误时，`error` 返回错误信息，`answer` 可同时提供适合用户阅读的提示。Agent 核心保持无状态，不在内部重复保存前端会话。

### 3. 过程展示边界

前端可以展示：

- 工具名称；
- 调用参数；
- 成功或失败状态；
- 关键结果摘要；
- 最终回答。

前端不展示模型隐藏思维链、完整 Prompt、密钥、环境变量或未经处理的内部日志。

## 四、测试与验收

三人分别测试自己的模块，并共同完成系统联调。

- 郑梓湃：覆盖直接回答、单工具、多工具、多轮上下文、工具失败、模型/API 失败和输出契约；
- 李文杰：覆盖工具集的正常查询、无数据、非法参数和数据源失败；
- 田佳瑛：覆盖对话交互、历史记录、过程展示、错误展示和端到端流程；
- 田佳瑛负责汇总不少于 10 条系统测试用例、测试结果和缺陷记录；
- Agent 测试采用 Mock 自动测试加少量真实 API 冒烟测试，避免只依赖网络和模型随机性。

最终版本至少满足：

1. 新环境可按 README 启动；
2. 基础工具及已实现扩展工具可独立验证；
3. 前端能通过统一入口完成完整问答；
4. 工具调用过程可见但不暴露内部敏感信息；
5. 测试报告、缺陷记录和个人贡献证据完整。

## 六、GitHub 协作规则

- 每人使用自己的 Git 身份和个人功能分支；
- 通过 Pull Request 合并到主分支，不由一人代替其他成员统一提交；
- 每次提交只包含一个明确任务，提交信息说明实际变更；
- 接口变更必须在 PR 中说明，并通知接口上下游成员；
- 郑梓湃负责合并前的接口检查和主分支可运行性；
- 不提交 `.env`、API Key、密钥或其他敏感配置。
