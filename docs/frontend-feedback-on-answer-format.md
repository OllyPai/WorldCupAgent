# 前端给后端的回答格式说明

本文档用于说明前端这边对后端 `answer` 字段展示的一项调整，以及我们这么做的目的。

## 1. 这次改了什么

前端联调时发现，当工具查询失败时，后端主回答 `answer` 中会直接出现类似下面的句子：

```text
query_schedule 调用失败：未查询到符合条件的赛程信息，请检查关键词是否正确。
```

这个句子里的 `query_schedule` 是后端内部工具名。  
它适合出现在调试信息或 Trace 面板里，但不适合直接展示给普通用户。

因此，这次我们在 `agent.py` 中把失败回答的主文案做了用户向收口：

- 原来：`query_schedule 调用失败：...`
- 现在：`赛程查询失败：...`

同理：

- `query_player_stats` -> `球员数据查询失败`
- `query_match_detail` -> `比赛详情查询失败`

## 2. 具体改动位置

文件：

- `agent.py`

改动点：

1. 新增 `_format_tool_error(call)`  
2. `_format_tool_answer(tool_calls)` 在 `status == "error"` 时不再直接拼接工具名，而是调用 `_format_tool_error(call)`

核心目的是把后端内部实现名和面向用户的展示文案分开。

## 3. 我们为什么要这样改

前端页面分两层展示：

1. `answer`
   用于主对话区 / 结果区，面向普通用户

2. `tool_calls`
   用于 Trace 面板，展示工具调用过程，面向演示和调试

如果把内部工具名直接放进 `answer`，会出现两个问题：

1. 用户会看到 `query_schedule` 这类技术词，理解成本高
2. 主回答和 Trace 面板职责重复，界面会显得像开发调试页面

因此我们希望：

- `answer` 保持自然语言、用户口吻
- `tool_calls` 保留真实工具名、输入参数、状态和摘要

## 4. 这次改动不影响什么

这次改动**不改变前后端接口契约**，只调整 `answer` 的文案内容。

没有变化的部分：

- `POST /api/chat` 请求结构不变
- 返回字段不变：`answer / tool_calls / error / result_payload`
- `tool_calls[].tool` 仍然保留原始工具名
- `tool_calls[].input`、`tool_calls[].status`、`tool_calls[].summary` 结构不变
- 前端 Trace 面板仍然可以正常显示真实工具调用过程

## 5. 前端当前的展示原则

前端当前约定如下：

- `answer`：主内容，用户可直接阅读
- `tool_calls`：过程信息，放在 Trace 面板
- `error`：错误提示

也就是说：

- 对用户：尽量不暴露内部实现名
- 对开发和演示：保留工具调用过程可见

## 6. 给后端同学的建议

后续如果继续优化 `answer`，建议优先保持以下原则：

1. 不把内部工具名直接写进主回答
2. 不把调试口吻文案直接给用户看
3. 主回答尽量写成自然语言
4. 调试和工具过程信息继续放在 `tool_calls`

更理想的失败文案风格例如：

- `没有找到符合条件的赛程，请换个球队、日期或比赛阶段再试。`
- `暂时没有查到这位球员的数据，请检查姓名是否正确。`
- `没有找到这场比赛的详情，请确认球队名称是否准确。`

## 7. 总结

这次调整的本质是：

- 不改接口
- 不改工具过程字段
- 只把 `answer` 从“开发者口吻”改成“用户口吻”

这样可以让前端页面更像正式产品界面，而不是调试界面。
