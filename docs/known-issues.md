# 当前已知问题与后续修复清单

记录时间：2026-07-07

更新：2026-07-08，补充 Agent 特殊输入与聚合查询能力问题。

本文档汇总当前版本中已经发现、但暂未立即修复的问题。当前版本可作为阶段性演示版本，但距离最终交付仍需要继续打磨界面表达、联调细节和正式文档。

## 1. 前端界面表达问题

### 1.1 结果区标题暴露开发状态

当前真实查询后，结果区在 `result_payload === null` 时会显示：

```text
当前接口暂未返回结构化结果
```

问题：

- 这句话是开发者视角，不适合出现在用户界面或答辩演示中；
- 容易让老师认为功能未完成；
- 实际情况是后端已经返回 `answer` 和 `tool_calls`，只是暂未返回卡片/表格用的 `result_payload`。

建议：

- 改为“查询结果”或“工具查询结果”；
- 说明文字改为“以下结果由后端智能体调用工具后生成”。

### 1.2 结果区说明仍像半成品

当前说明包含类似：

```text
现阶段以后端返回的文本 answer 为主，后续再逐步补充卡片或表格。
```

问题：

- 属于开发过程说明；
- 不适合放在面向用户/老师的正式界面。

建议：

- 从 UI 中移除“现阶段”“后续再补”等表述；
- 如果需要说明数据来源，使用“结果来自本地课程演示数据库”。

### 1.3 页脚团队信息未完成

当前页脚仍可能显示：

```text
团队：待补充
数据：用于界面展示的模拟数据
```

问题：

- “团队：待补充”明显未完成；
- “模拟数据”表述不够准确，当前系统包含两类数据：
  - 后端查询数据：`tools/worldcup.db` 本地 SQLite 课程演示数据库；
  - 首页展示数据：前端演示样例。

建议：

- 补全团队成员：郑梓湃、李文杰、田佳瑛；
- 将数据说明改为“本地 SQLite 课程演示数据库 + 前端演示样例”。

### 1.4 UI 中仍有开发者语气

当前部分文案类似：

```text
这部分是你前端的加分核心
后续再逐步补充
当前接口暂未返回
```

问题：

- 这些话是写给开发者看的，不是写给用户或评审看的；
- 答辩时容易暴露项目仍处于草稿状态。

建议：

- 全局清理开发者语气；
- 改为用户视角和系统能力说明，例如“这里展示本次回答涉及的工具名称、输入参数和执行状态”。

## 2. Trace 面板状态问题

### 2.1 未查询时显示“成功”

当前未发送问题时可能显示：

```text
调用次数：0
整体状态：成功
```

问题：

- 没有调用工具时不能算成功；
- 会误导用户和评审。

建议：

- `tool_calls.length === 0 && !error` 时显示“待查询”；
- 只有收到实际工具调用结果后才显示“成功/失败”。

### 2.2 未查询时仍展示固定流程

当前未发送问题时，Trace 面板仍可能展示：

```text
接收自然语言问题
将 history 与 user_input 发送给智能体
调用世界杯工具集
整合工具结果并生成回答
```

问题：

- 这些步骤在未查询时尚未发生；
- 容易把“系统流程说明”误显示成“本次执行轨迹”。

建议：

- 未查询时显示空状态：“发送问题后将在这里展示工具调用过程”；
- 查询后再展示实际 `tool_calls` 和流程。

## 3. Mock / 示例数据边界问题

### 3.1 首页仍是静态演示数据

当前首页数据来自：

```text
frontend/src/data/homeMock.js
```

问题：

- 首页不是从后端实时查询；
- 虽然已改为“本地演示数据”口径，但本质仍是静态样例。

建议：

- 短期：继续明确标注“本地演示数据”；
- 中期：如有时间，可新增后端首页数据接口；
- 答辩时说明首页用于展示产品形态，查询页才是 Agent 真实调用入口。

### 3.2 查询页仍使用 `queryMock.js` 管理示例

当前查询页真实提交会调用后端 `/api/chat`，但示例按钮和预设展示仍由：

```text
frontend/src/data/queryMock.js
```

管理。

问题：

- 文件名 `queryMock.js` 容易让人误解为查询结果仍是 Mock；
- 实际它现在更接近“示例问题/示例卡片配置”。

建议：

- 后续可重命名为 `queryExamples.js`；
- 清理未使用的 mock payload 生成函数；
- 保留“示例问题”功能，但避免伪造真实回答。

### 3.3 `result_payload` 尚未由后端生成

当前后端 API 固定返回：

```json
"result_payload": null
```

问题：

- 前端结构化卡片区无法直接展示真实后端结构化数据；
- 真实查询主要依赖 `answer` 和 `tool_calls`。

建议：

- 短期：UI 只把 `answer` 作为查询结果展示；
- 中期：如果需要表格/卡片，可在后端根据工具结果生成 `result_payload`；
- 文档中继续明确 `result_payload` 是可选字段。

## 4. Agent 与工具能力问题

### 4.1 特殊输入可能得到错误回答

当前在某些特殊输入下，Agent 可能给出不符合预期的回答。

可能场景包括：

- 输入内容不属于世界杯赛事查询；
- 输入内容过短、语义不完整或带有异常符号；
- 用户提出当前工具集无法支持的问题；
- 模型没有正确识别应直接提示能力边界，还是应调用工具。

问题：

- 用户会认为系统“答非所问”；
- 答辩时如果老师输入边界问题，系统表现不稳定；
- 当前测试更多覆盖标准问题，对特殊输入覆盖不足。

建议：

- 后端补充输入分类或兜底策略；
- 对无法支持的问题，明确返回“当前系统主要支持赛程、球员数据、比赛详情查询”；
- 补充特殊输入测试用例，例如空白、乱码、非世界杯问题、超出工具范围的问题；
- 前端可在示例区提示系统支持的问题类型，降低用户输入偏离范围的概率。

归属：

- 郑梓湃负责 Agent 兜底逻辑与测试；
- 田佳瑛负责前端支持范围提示；
- 李文杰无需改工具，除非问题最终确认和数据查询有关。

### 4.2 无法回答“某国进球最多的球员”

当前用户询问类似：

```text
巴西进球最多的球员是谁？
法国队进球最多的球员是谁？
```

系统无法稳定给出答案。

原因：

- 当前 `query_player_stats` 只支持按球员姓名查询；
- 工具集中没有“按球队查询球员列表并按进球排序”的能力；
- LLM 不能直接从数据库聚合数据，只能调用已暴露的工具。

问题：

- 这类问题属于自然的世界杯数据查询，用户很可能会问；
- 如果模型绕过工具直接回答，容易产生幻觉；
- 如果模型调用现有球员查询工具，又缺少具体球员姓名，查询会失败。

建议：

- 新增工具 `query_top_scorer_by_team(team: str)`，返回该球队进球最多球员；
- 或扩展现有工具，支持 `team` 参数和排序返回；
- 工具返回字段至少包括：球队、球员姓名、进球、助攻、出场次数；
- 增加测试用例：“查询某队进球最多球员”“查询不存在球队进球最多球员”。

归属：

- 李文杰负责工具与数据库查询实现；
- 郑梓湃负责将新工具注册到 Agent，并补充工具选择测试；
- 田佳瑛负责前端示例问题和展示文案更新。
