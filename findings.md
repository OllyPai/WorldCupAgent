# WorldCupAgent 发现与决策

## 项目需求

- 课程题目为“基于大模型的工具调用智能体”，当前产品场景为世界杯竞赛播报助手。
- 系统需要体现“理解任务 → 选择工具 → 调用工具 → 观察结果 → 生成回答”的完整流程。
- 必做工具是赛程查询、球员数据查询和比赛详情查询。
- 必须覆盖直接回答、单工具、多工具、多轮对话、错误输入与异常提示。
- 需要 Web 可视化界面和不少于 10 条测试用例。
- 过程文档、测试报告、个人贡献记录和答辩演示会影响课程评分。

## 团队与接口

- 郑梓湃负责 Agent 核心、统一入口、集成、配置和系统设计说明书。
- 李文杰负责三个工具、数据来源、工具契约和工具测试。
- 田佳瑛负责 Web 界面、历史维护、过程展示和测试汇总。
- 工具统一返回：

```python
{"success": True, "data": {}, "error": None}
```

- Agent 面向前端统一返回：

```python
{"answer": "...", "tool_calls": [], "error": None}
```

- 前端维护历史，Agent 核心保持无状态。

## 已知质量问题

- 梅西查询中，模型曾在工具只返回出场数和进球数时，额外补充参赛届数、夺冠和金球奖等事实。
- 赛程和比赛详情真实冒烟中，模型也曾推断“小组头名出线”“晋级下一轮”。
- 这说明“调用了正确工具”不等于“最终回答严格基于工具结果”；当前已改为工具型问题由代码格式化最终回答。
- PR #2 数据应定位为本地 SQLite 课程演示数据库，不应在答辩或文档中声明为官方实时数据。
- 当前 10 条测试是 Agent 核心测试，不等于最终完整系统测试。
- `agent.py` 仍是单文件学习版；当前不做大规模拆分。

## 技术决策

| 决策 | 理由 |
|------|------|
| Python 3.12 + `.venv` | 与当前环境一致，并隔离依赖 |
| LangChain `create_agent` | 使用现成工具调用循环，符合学习目标和工期 |
| `langchain-deepseek` | 使用模型提供方对应集成包 |
| `deepseek-v4-flash` | 当前代码已验证可用 |
| 依赖注入 `agent=` 仅用于测试 | 无需真实网络即可验证统一入口 |
| 暂不拆分复杂目录结构 | 先完成稳定行为，再按真实集成需要拆分 |
| 当前接入 PR #2 工具但不直接合并 main | 先验证工具接口能被 Agent 使用，再决定是否在 GitHub 合并 PR |
| LLM 负责选工具，代码负责格式化工具型最终回答 | 保证最终答案只包含工具返回字段，避免模型自由发挥 |
